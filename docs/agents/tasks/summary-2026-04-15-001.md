## What changed
- Fixed a ParsePage concurrency bug by replacing the shared per-language `FwDict` cache with a thread-safe nested concurrent dictionary.
- Added a ParsePage regression test that runs language replacements in parallel to verify the cache no longer throws or corrupts translations.
- Guarded missing-translation file updates so `langMap` does not try to write `lang/<lang>.txt` when no template root is configured.

## Commands that worked (build/test/run)
- `run_tests` for `osafw.Tests.ParsePageTests`
- `run_build`

## Pitfalls - fixes
- The outer language cache was concurrent, but each language entry stored a mutable `FwDict`; concurrent parser requests could mutate it at the same time and trigger `InvalidOperationException`.
- Focused tests exposed an existing no-template-root edge case in `langMap`; added a `TMPL_PATH` guard so tests and utility usage remain safe.

## Decisions - why
- Kept the fix localized to `ParsePage` instead of changing `FwDict` globally because the failure was caused by one shared cache instance, not by all dictionary usage across the framework.
- Reused the existing shared cache design so language files still load once per language while gaining safe concurrent reads/writes.

## Heuristics (keep terse)
- When a `ConcurrentDictionary` stores mutable inner collections shared across requests, make the nested collection concurrent too or synchronize all access.
