How To Use This Page

This page demonstrates project-specific choices. It does not replace Bootstrap documentation; it shows the conventions layered on top of Bootstrap in this repository.

  1. Start with Bootstrap classes and utilities for common layout and controls.
  2. Use framework classes for repeated app patterns: page headers, cards, filters, list tables, fieldsets, empty states, and attachments.
  3. Change look and feel through CSS custom properties before adding selector-specific overrides.
  4. Keep custom themes compatible with Bootstrap variables so Bootstrap-based plugins continue to fit.
  5. Document gaps here before changing shared CSS, especially if a theme-specific value appears unused or duplicated.

Design Principles

The visual system is intentionally utilitarian: dense enough for repeated business workflows, clear enough for developers to start without a dedicated designer.

Bootstrap First

Use Bootstrap 5 components, variables, and utilities as the baseline. The framework layer should stay small and compatible with Bootstrap upgrades.

Token First

Shared visual decisions live in custom properties such as --fw-pane-bg, --fw-card-radius, and --fw-btn-font-size.

Data Density

Lists, filters, and forms optimize for scanning and repeated actions. Avoid decorative layouts that reduce usable space on admin screens.

Themeable By Default

Light and dark modes share the same component markup. Optional themes override Bootstrap and framework variables after site.css.

Customization Guidance

Use the smallest layer that fits the change. This keeps generated modules consistent while allowing product-specific branding.

Load Order

  1. Bootstrap CSS
  2. Bootstrap Icons
  3. site.css
  4. Optional themeXX.css
  5. Page-specific head.css when needed

Preferred Override Order

  1. Bootstrap utilities for one-off layout tweaks.
  2. Framework tokens for shared visual behavior.
  3. Theme tokens for project branding.
  4. Component selectors only when tokens cannot express the need.

Theme Hooks

Layouts set data-fw-theme and data-bs-theme. Use [data-bs-theme="dark"] for mode changes and keep theme ids stable for project-specific overrides.

Generated Modules

Generated list, view, and edit screens already use shared fragments. Prefer adjusting config.json, shared slots, or theme tokens before cloning markup.
